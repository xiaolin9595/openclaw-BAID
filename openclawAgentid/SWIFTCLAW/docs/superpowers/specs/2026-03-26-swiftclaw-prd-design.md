# SWIFTCLAW PRD (Product Requirements Document)

**设计文档** | 版本: v1.0 | 日期: 2026-03-26

---

## 1. 产品概述

### 1.1 产品定位

**SWIFTCLAW** 是一个面向个人用户的开源 AI 助手平台，采用 Python 技术栈实现。它运行在用户自己的设备上，通过 Telegram 提供私密的 AI 助手服务，具备工具调用、持久化记忆和安全的代码执行能力。

### 1.2 核心价值主张

- **隐私优先**: 数据保存在本地，不依赖第三方云服务
- **工具丰富**: 支持文件操作、Bash 命令、浏览器自动化
- **记忆持久**: 跨会话记住用户偏好和历史上下文
- **安全可靠**: Docker 沙箱隔离危险操作
- **易于部署**: 单体架构，一行命令启动

### 1.3 目标用户

- 技术爱好者（程序员、极客）
- 注重隐私的个人用户
- 希望拥有私有化 AI 助手的 OpenClaw 替代者

## 2. 功能需求

### 2.1 核心功能模块

| 模块 | 功能 | 优先级 |
|------|------|--------|
| **Gateway** | FastAPI WebSocket/HTTP 服务 | P0 |
| **Telegram 通道** | Bot 消息收发、命令解析 | P0 |
| **Agent 运行时** | Moonshot API 调用、流式输出 | P0 |
| **工具系统** | 文件、Bash、浏览器工具 | P0 |
| **记忆系统** | SQLite 持久化存储 | P1 |
| **搜索工具** | Tavily Search 联网搜索 | P1 |
| **安全沙箱** | Docker 隔离执行 | P1 |

### 2.2 详细功能规格

#### 2.2.1 Gateway 网关

**需求**:
- WebSocket 长连接支持实时双向通信
- HTTP API 用于健康检查、状态查询
- 自动重连机制
- 请求日志记录

**接口**:
```
WS   /ws/agent          # Agent 实时对话
GET  /health            # 健康检查
GET  /api/status        # 服务状态
POST /api/message       # 发送消息（HTTP 备选）
```

#### 2.2.2 Telegram 通道

**需求**:
- 长轮询接收消息
- 支持文本消息和命令（/start, /help, /clear）
- 流式响应分片发送（避免消息过长）
- 用户身份验证（白名单机制）

**命令**:
- `/start` - 开始使用，显示欢迎信息
- `/help` - 显示可用命令和工具
- `/clear` - 清空当前会话记忆
- `/status` - 查看服务状态

#### 2.2.3 Agent 运行时

**需求**:
- 基于 PydanticAI 调用 Kimi API
- 支持多模型切换（moonshot-v1-8k/32k/128k）
- 流式输出支持打字机效果
- 工具调用循环（Tool Use Loop）
- 系统提示词可配置

**工具调用流程**:
```
用户输入 → LLM 判断是否需要工具 → 执行工具 → 结果返回 LLM → 生成回复
```

#### 2.2.4 工具系统

**文件工具** (`file_read`, `file_write`, `file_list`)
- 读取文本文件（支持编码自动检测）
- 写入文件（支持追加模式）
- 列出目录内容
- 限制：只能在指定工作目录内操作

**Bash 工具** (`bash_execute`)
- 执行 shell 命令
- 超时控制（默认 30s，可配置）
- 输出限制（stdout/stderr 截断）
- Docker 沙箱隔离执行

**浏览器工具** (`browser_open`, `browser_click`, `browser_extract`)
- 打开指定 URL
- 截图保存
- 点击元素
- 提取页面文本内容

**搜索工具** (`web_search`)
- Tavily Search API 集成
- 搜索结果摘要
- 引用来源链接

#### 2.2.5 记忆系统（SQLite）

**需求**:
- 用户偏好存储（语气风格、常用指令）
- 对话历史保留（最近 N 轮）
- 关键事实记录（用户提到的信息）
- 跨会话持久化

**数据表设计**:
```sql
-- 用户表
users (id, telegram_id, created_at, preferences)

-- 对话历史表
conversations (id, user_id, role, content, timestamp, session_id)

-- 关键记忆表
memories (id, user_id, key, value, importance, created_at)
```

#### 2.2.6 安全沙箱（Docker）

**需求**:
- Bash 命令在 Docker 容器中执行
- 文件系统隔离（只挂载指定目录）
- 网络隔离（可选）
- 资源限制（CPU、内存）
- 超时强制终止

## 3. 非功能需求

### 3.1 性能要求

| 指标 | 目标 |
|------|------|
| 首次响应时间 | < 3s |
| 流式输出延迟 | < 100ms/字符 |
| 并发用户数 | 5（个人使用）|
| 启动时间 | < 5s |

### 3.2 安全要求

- API 密钥存储在环境变量，不硬编码
- 用户白名单机制（拒绝未授权用户）
- 文件操作限制在工作目录
- Bash 命令 Docker 隔离
- 敏感日志脱敏

### 3.3 可靠性要求

- 异常捕获和优雅降级
- 自动重连机制
- 定期备份 SQLite 数据库
- 健康检查端点

### 3.4 开发规范

- **SDD + TDD**: 所有功能必须遵循 Spec-Driven Development 和 Test-Driven Development
- **类型注解**: 100% 类型覆盖
- **测试覆盖**: 核心功能 > 80%
- **代码风格**: ruff + black

## 5. 开发路线图

### Phase 1: 核心框架（Week 1-2）

**目标**: 搭建基础架构，实现最简端到端流程

- [ ] 项目脚手架（FastAPI + Telegram Bot）
- [ ] Moonshot API 集成（pydanticAI）
- [ ] 基础对话流程
- [ ] 环境配置管理

**验收标准**:
- 能通过 Telegram 发送消息，收到 Kimi 回复
- 流式输出正常工作

### Phase 2: 工具系统（Week 3-4）

**目标**: 实现核心工具

- [ ] 文件工具（读/写/列表）
- [ ] Bash 工具（带 Docker 沙箱）
- [ ] 浏览器工具（Playwright）
- [ ] 工具调用循环

**验收标准**:
- "帮我读取 ./test.txt" → 成功返回内容
- "执行 ls -la" → 在沙箱中执行并返回结果
- "打开 https://example.com" → 截图或提取内容

### Phase 3: 记忆与搜索（Week 5-6）

**目标**: 持久化记忆和联网能力

- [ ] SQLite 数据库设计
- [ ] 对话历史存储
- [ ] 用户偏好记忆
- [ ] Tavily Search 集成

**验收标准**:
- 重启后能记住用户偏好
- "搜索 Python 最新版本" → 返回搜索结果

### Phase 4: 完善与优化（Week 7-8）

**目标**: 提升稳定性和体验

- [ ] 错误处理优化
- [ ] 性能优化
- [ ] 测试覆盖提升
- [ ] 文档完善

**验收标准**:
- 单元测试覆盖率 > 80%
- 异常情况下优雅降级

## 6. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Docker 沙箱配置复杂 | 高 | 中 | 提供一键脚本，文档详细说明 |
| Kimi API 限制 | 中 | 低 | 通过 litellm 支持多模型备份 |
| Telegram 长轮询不稳定 | 中 | 中 | 实现重连机制，支持 Webhook 备选 |
| 并发性能瓶颈 | 中 | 低 | 明确单用户定位，后续可扩展 |
