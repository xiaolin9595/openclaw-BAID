# AgentID 身份控制台网站设计

状态：Draft  
日期：2026-07-10  
关联文档：[AgentID 与 OpenClaw Instance 授权绑定设计](./agentid-openclaw-instance-authorization-design.md)

## 1. 定位

AgentID 网站是用户创建、拥有和治理 Agent 的身份控制台。它的核心问题不是展示区块链、模型或网络指标，而是回答三个问题：

1. 我拥有哪些 Agent？
2. 哪些设备被允许代表每个 Agent 运行？
3. 最近发生了哪些影响身份和权限的事件？

网站不直接运行 Agent，不持有 OpenClaw 实例私钥，也不要求普通用户理解 `instanceId`、`peerId`、IBC 或 DHT。

## 2. 目标与非目标

### 2.1 目标

- 新用户可在注册后快速创建第一个 Agent。
- 已登录用户可通过扫码或短码，在 60 秒内完成一个 OpenClaw 实例授权；需要登录或 Passkey 二次确认的用户仍应保留完整请求上下文。
- 用户可清楚地查看、续期和撤销每个 Agent 的设备授权。
- 身份相关动作有可追溯审计记录，且高风险动作要求额外确认。
- 常规管理体验保持简洁，技术身份信息仅在需要时展开。

### 2.2 非目标

- 不做面向访客的营销落地页作为产品首屏。
- 不在一期加入钱包、区块链交易、ZK-KYC、模型监控或任务编排。
- 不在普通页面显示用户内部 ID、完整凭证、私钥、`device_code` 或令牌；授权页可显示短期 `user_code` 供用户核对。
- 不通过浏览器端 `localStorage` 保存授权事实或安全状态。

## 3. 用户与权限

| 角色 | 典型需求 | 可执行操作 |
| --- | --- | --- |
| Owner | 创建和控制自己的 Agent | 全部操作，包括成员、策略、授权和撤销 |
| Admin | 协助管理 Agent | 授权、续期、撤销设备，管理非 Owner 成员 |
| Operator | 维护运行设备 | 查看设备与活动，发起续期请求 |
| Viewer | 审阅状态 | 只读查看 Agent、设备和活动 |

个人版默认只有 Owner。只有在成员协作真实出现后，才显示成员管理入口。

## 4. 信息架构

登录后的主导航固定为四项：

```text
我的 Agent
待授权
活动记录
账户与安全
```

`待授权` 是独立一级入口：用户不应先找到某个 Agent 才能处理客户端发起的授权请求。

路由建议：

```text
/login
/register
/agents
/agents/new
/agents/:agentId
/agents/:agentId/instances/:instanceId
/approvals
/device                         # 二维码进入的独立安全页
/activity
/settings/security
```

`/device` 不展示侧边栏、Agent 列表或任何分散注意力的导航，只处理当前一个设备授权请求。请求通过不透明的服务端会话标识解析；页面、访问日志、分析事件和 Referrer 中不得记录 `device_code`。

## 5. 核心用户流程

### 5.1 注册与创建第一个 Agent

1. 用户用 Passkey、密码或第三方登录注册 AgentID。
2. 系统进入“没有 Agent”的空状态，而不是通用仪表板。
3. 用户点击“创建 Agent”，填写名称和可选用途说明。
4. 系统创建 `agentId` 并将当前用户写为 Owner。
5. 成功页直接进入该 Agent 的工作区，主操作是“连接 OpenClaw”。

空状态文案：

```text
还没有 Agent

创建一个 Agent 后，你可以授权 OpenClaw 在不同设备上运行它。

[创建 Agent]
```

### 5.2 授权 OpenClaw 实例

客户端调用设备码授权接口后展示二维码与短码。用户在同一或另一设备打开 `/device`：

1. 登录网站，或复用已有登录状态。
2. 页面校验并展示待授权实例的信息。
3. 系统优先选中客户端提交的 `agent_hint`；用户仅可在自己有 `instance:approve` 权限的 Agent 之间显式切换。
4. 用户核对页面短码和客户端短码，并勾选“我确认两处代码一致”。
5. 用户点击“授权设备”；首次绑定、策略要求或高风险权限要求 Passkey 二次确认。
6. 服务端以原子方式将请求标记为已批准并签发 IBC；并发批准时仅第一个有效，其余操作显示“请求已处理”。
7. 客户端轮询兑换结果；页面显示成功状态，用户返回客户端。

### 5.3 日常设备治理

1. 用户在 Agent 工作区的“设备”标签查看实例列表。
2. 用户可查看设备状态、到期时间、最近活动与技术详情。
3. 用户可撤销某个设备；撤销是不可逆安全动作，要求二次确认。
4. 临近过期时，系统在 Agent 页面和待办区域提示续期。

## 6. 页面设计

### 6.1 登录与注册

页面只承载登录，不与营销内容混排。首选操作是“使用 Passkey 继续”；密码和第三方登录放在同一层级的次级入口。

注册成功后不要求用户填写证件、钱包或复杂身份资料。若未来需要组织或合规验证，应在明确的业务动作中渐进收集。

### 6.2 我的 Agent

页面采用紧凑列表，不使用统计卡片瀑布流。每行是一个 Agent：

| Agent | 设备 | 待处理 | 最近活动 | 操作 |
| --- | --- | --- | --- | --- |
| 旅行助理 | 2 台设备，1 台在线 | 无 | 2 分钟前设备续期 | 打开 |
| 研发助手 | 1 台设备，离线 | 1 个请求 | 刚刚收到授权请求 | 处理 |

顶部操作只有“创建 Agent”。空状态与列表状态使用同一个页面，不跳转到单独的欢迎页。

### 6.3 Agent 工作区

这是网站的对象页，而不是仪表盘。顶部提供稳定的身份摘要：

```text
旅行助理                                             已保护
2 台已授权设备 · 0 个待处理请求            [连接设备] [管理]
```

使用四个 Tab：

- 设备：默认页，展示所有已授权实例。
- 成员：多人协作启用后展示。
- 规则：授权期限、离线容忍期和高风险动作策略。
- 活动：该 Agent 的审计记录。

设备列表建议使用表格行：

| 设备 | 状态 | 凭证 | 最近活动 | 操作 |
| --- | --- | --- | --- | --- |
| Lin's MacBook | 在线 | 89 天后到期 | 2 分钟前 | 查看，撤销 |
| Production Server | 离线 | 42 天后到期 | 昨天 | 查看，撤销 |

普通用户看到“设备”。`instanceId`、`peerId`、IBC 编号和公钥指纹在“技术详情”折叠区显示，且都支持复制。

### 6.4 待授权

待授权页按风险和有效期排序。每项明确显示：目标 Agent、设备名称、平台、请求时间、剩余时间、短码和权限范围。页面在前台时每 10 秒刷新；用户点击批准或拒绝后立即重新读取服务端状态。请求列表和活动记录均使用 cursor 分页。

对 Owner/Admin 提供“授权”和“拒绝”；其他角色只能查看状态。请求过期后保留审计记录，但不再显示可操作按钮。

### 6.5 设备授权安全页

`/device` 是独立的全屏页面，视觉上应像一张安全收据：无侧栏、无统计模块、无推广入口。

```text
确认新设备

旅行助理
正在请求连接一台 OpenClaw 设备

Lin's MacBook
macOS · 请求有效期剩余 08:42

确认码      FJ8K-M4QN
密钥指纹    7A:31:...:C4
权限        发送消息、发布网络状态
凭证        有效期 90 天，可随时撤销

[拒绝请求]                              [授权设备]
```

规则：

- 页面提供“我已确认两处代码一致”的复选框；未勾选时禁用批准按钮。网站无法自行判断用户屏幕上的代码是否一致。
- 页面说明“只批准你刚刚在自己的 OpenClaw 客户端发起的请求”，降低设备码钓鱼风险。
- 已过期、已拒绝、已兑换或密钥变化的请求显示单独状态，不能复用成功页。
- 发生 Passkey 二次验证后，回到原请求而非 Agent 列表。
- 授权成功页只说明“设备已授权，可返回客户端”，不展示任何令牌或完整凭证。

### 6.6 实例详情与撤销

实例详情分为“概览”“技术详情”“活动”三段。撤销按钮置于页面末尾的危险区域，使用红色文字按钮和二次确认对话框。

撤销对话框必须说明影响：该设备将无法继续代表此 Agent 通信，重新启用需要客户端发起新的授权请求。对话框要求用户确认设备名，避免误撤销相近名称设备；除非存在近期的高保证登录，否则提交撤销还需要 Passkey 二次确认。

### 6.7 活动记录与账户安全

活动记录以时间线展示身份动作：创建 Agent、批准实例、拒绝请求、续期、撤销、成员和策略变更。每项包含执行者、对象、时间和结果。

账户与安全只展示与账户保护直接相关的内容：Passkey、活跃登录会话、恢复方式和个人资料。不要把 Agent 管理配置塞入该页面。

### 6.8 Agent 属性公开展示

本次新增范围只围绕 Agent 属性展示，不新增独立的用户公开主页。用户个人网页或 Agent 工作区中增加一个“公开属性”区域；该区域可以被访客看到，但只展示用户明确标记为公开的 Agent 信息。

建议路由：

```text
/agents/public                       # 未登录可访问的 Agent 公开目录，支持搜索和筛选
/agents/:agentId                     # 登录后的 Agent 管理页，包含公开属性预览
/agents/:agentId/public              # Agent 公开详情页
```

公开目录是访客进入 AgentID 网站后的默认入口，不直接把某一个 Agent 作为首页。目录包含搜索框、已验证筛选、公开 Agent 数量、Agent 卡片和分页/加载更多。卡片只展示名称、简介、公开属性摘要、身份验证状态和实例数量；点击卡片后进入对应的 Agent 公开详情页。

目录的高级发现交互参考现有 Agent Discovery 实现，但只保留适合公开展示的字段：

- 精选 Agent 区域：展示公开属性完整、身份已验证的 Agent。
- 左侧筛选：身份状态、Agent 角色、主要能力、运行环境。
- 搜索：支持名称、简介、公开能力、公开标签和 AgentID；生产环境由服务端分页和索引完成。
- 排序：综合排序、公开连接数、公开评分、公开响应时间；没有可靠来源的数据不展示。
- 视图：网格卡片适合浏览，列表视图适合比较多个 Agent。
- 卡片摘要：身份状态、角色、语言/运行环境、能力标签、公开评分、公开连接数和稳定性摘要。

评分、连接数、响应时间和稳定性属于可选公开运行指标，必须明确数据来源和更新时间，不能把用户账户数据或未验证的在线状态伪装成服务端事实。Demo 可以用固定演示数据展示信息层级，接入真实 API 后由公开 DTO 提供。

目录搜索范围只包括公开 DTO 中的 Agent 名称、简介、公开标签、公开能力和 AgentID。用户 ID、邮箱、完整 IBC、设备公钥、JTI 和成员关系不能进入搜索索引或目录卡片。

公开资料只描述 Agent，不展示用户个人账户资料：

```text
Research Assistant
AgentID: did:agentid:agt_xxx
负责论文检索、摘要和资料整理

公开能力：research · summarization
公开标签：academic · P2P
公开项目：open-research
身份状态：已验证
状态：未知
```

Agent 属性分为：

| 属性 | 示例 | 默认可见性 | 说明 |
|---|---|---|---|
| Agent 名称 | `Research Assistant` | 公开 | Agent 的可读名称 |
| Agent 简介 | `负责论文检索` | 公开 | 用户主动填写 |
| 标签 | `academic`、`P2P` | 公开 | 对应 `userPublicAttributes` 的 tag |
| 能力 | `skill=research` | 公开 | 默认标记为“自声明” |
| 角色 | `role=assistant` | 可选公开 | 只展示用户明确公开的角色 |
| 项目/群组 | `project=open-research` | 可选公开 | 不默认暴露组织内部信息 |
| 身份验证状态 | `已验证` | 公开 | 只显示摘要，不显示完整 IBC |
| 在线状态 | 在线/最近活跃/未知 | 可选公开 | 必须有可靠 heartbeat 或发现数据 |

能力字段必须标注信任级别：

```text
自声明能力：skill=research
身份已验证：AgentID/InstanceID 绑定有效
能力已验证：由指定能力签发者或测试流程签名确认
```

AgentID/IBC 的身份验证不等于能力验证。公开页面不能把自声明的 `skill` 展示成服务端背书的权限。

### 6.9 Agent 属性管理交互

1. Owner/Admin 在 Agent 工作区打开“公开属性”编辑区。
2. 用户编辑名称、简介、标签、能力、项目和可选状态摘要。
3. 每项属性使用公开/不公开开关；保存前显示访客预览。
4. 服务端保存公开属性和可见性，不允许前端只修改本地状态。
5. Agent 公开页只读取公开 DTO；不能通过参数切换读取管理字段。

### 6.10 从发现到通信

公开页面的职责是帮助人或第三方 Agent 找到目标，不承担 P2P 消息转发。详情页提供“发起通信”入口，第三方可以填写首条消息并复制最小调用参数：

```json
{
  "agentId": "did:agentid:agt_xxx",
  "message": "你好，我想了解你的研究能力。"
}
```

接入 OpenClaw 后，调用 `p2p_send_agent_message`：

```text
公开目录 / 详情页
  -> 搜索并确认目标 AgentID
  -> 复制 agentId 和消息参数
  -> 调用方 OpenClaw 根据 AgentID 查找已发现路由
  -> 仅保留 IBC 验证成功、未撤销、未过期的 Instance
  -> 使用实例私钥签名 p2p:message
  -> 对端验证 peer 信封、消息签名、IBC、AgentID、InstanceID 和 scope
  -> 返回 ACK 和投递结果
```

通信入口不展示用户 ID、邮箱、PeerID、JTI、完整 IBC 或私钥。公开页显示的“已验证”表示 Agent 身份绑定有效，不表示该 Agent 当前一定在线；实际发送还要求目标实例已经被本地 Mesh 发现。若未来需要“网页发起连接请求 -> 对方批准 -> 客户端异步建立通信”，应另增 `connection_requests` 服务端资源和客户端轮询命令，不应让浏览器直接执行 P2P 连接。
6. Viewer 可以查看允许公开的属性，但不能修改属性；Owner/Admin 才能编辑。
7. 修改属性后写入 `agent_updated` 或独立的 `agent_public_profile_updated` 审计事件。

公开属性的状态使用 `active`、`revoked`、`expired` 和 `unknown` 的用户可读表达；没有可靠 heartbeat 时显示“状态未知”，不能伪造在线状态。

## 7. 状态模型与异常体验

### 7.1 三条独立状态轴

授权请求、绑定凭证和网络连接不是同一状态，界面不得混用它们。

| 对象 | 状态 | 用户看到的表达 | 允许操作 |
| --- | --- | --- | --- |
| 授权请求 | `pending` | 待确认，显示倒计时 | 批准或拒绝 |
| 授权请求 | `approved` / `exchanged` | 已授权或已在客户端完成 | 查看活动 |
| 授权请求 | `denied` / `expired` | 已拒绝或已过期 | 查看活动；过期后需客户端重新发起 |
| 实例绑定 | `active` | 已授权 | 查看、按策略续期、撤销 |
| 实例绑定 | `expiring` | 即将到期，突出剩余天数 | 续期、撤销 |
| 实例绑定 | `expired` / `revoked` | 已失效或已撤销 | 查看活动；重新启用需新授权 |
| 网络连接 | `online` / `offline` / `unknown` | 在线、离线或状态未知 | 查看、撤销 |
| 验证结果 | `failed` | 验证失败，说明具体原因 | 查看详情，发起新授权 |

“离线”仅表示当前未连接，不等同于安全异常；“状态未知”表示最近心跳或发现数据不可用。不要使用无法指导操作的“置信度”指标。

### 7.2 关键状态转换

服务端是所有状态转换的唯一执行者。请求转换使用乐观锁或条件更新，确保 `pending` 只能被一次性转换为 `approved`、`denied` 或 `expired`；`approved` 之后仅允许一次 `exchanged`。撤销始终优先于续期：已撤销或已过期的绑定不能静默恢复。

续期规则为：实例先用现有私钥证明持有密钥，再请求新的 IBC。默认情况下，仍有效且密钥、权限和策略未变化的绑定可静默续期；以下情况必须重新由 Owner/Admin 批准：密钥变化、权限扩大、绑定已撤销、凭证已过期超出离线容忍期，或策略要求人工确认。

### 7.3 空状态和错误状态

- 无 Agent：引导创建 Agent。
- 无待授权请求：说明授权请求会在 OpenClaw 客户端发起后出现在这里。
- 网络错误：保留最近成功读取的数据和刷新按钮，避免整页空白。
- 权限不足：说明需要 Owner/Admin 批准，并显示请求管理员帮助的路径。
- 设备名重复：允许重复，但强制显示公钥指纹和最近活动以便区分。

## 8. 视觉与组件系统

设计方向为工业化安全控制台：冷静、清晰、具有可审计感。避免营销站的英雄区、渐变、悬浮装饰和大面积圆角卡片。

### 8.1 视觉令牌

| 类别 | 建议 |
| --- | --- |
| 背景 | 矿物白 `#F4F6F3`，表面白 `#FFFFFF` |
| 文字 | 石墨黑 `#18201D`，次级灰绿 `#5E6A64` |
| 可信 | 深青绿 `#147D72` |
| 待处理 | 深琥珀 `#8A5600` |
| 风险 | 深砖红 `#A13325` |
| 边框 | 浅灰绿 `#D9DFDA` |
| 字体 | `Noto Sans SC` 用于中文正文；`IBM Plex Mono` 用于 ID、短码和指纹 |

所有文字与背景组合在常规文字尺寸下至少达到 WCAG 2.2 AA 的 4.5:1 对比度；状态同时使用文字和图标，不只依赖颜色。圆角为 4-6px；只有对话框和二维码容器可使用 8px。图标使用统一的线性图标库，图标按钮必须提供 Tooltip 和无障碍名称。

### 8.2 组件规则

- 使用表格行、列表、分隔线和状态标签承载高频信息；不把每个数据项装进卡片。
- 主操作使用文字按钮，例如“连接设备”“授权设备”；工具操作使用图标按钮。
- 技术值使用等宽字体，并提供复制图标；默认截断，点击后展示完整值。
- 危险操作不使用红色实心主按钮；先用文字按钮，再在确认对话框中明确影响。
- 页面标题使用紧凑字号，只有登录或安全确认页允许更大的标题。

### 8.3 动效

动效仅服务于状态反馈：二维码等待时的轻微扫描线、授权成功后的状态切换、列表中新事件的短暂高亮。禁止连续装饰性动画和会改变布局的加载效果。

## 9. 响应式与可访问性

桌面端采用固定 224px 侧栏、主内容区和最大 1280px 阅读宽度。设备表格在窄屏转换为两行列表：第一行设备名和状态，第二行有效期与操作。

移动端优先保证 `/device`：二维码确认、短码、批准和拒绝按钮必须在单屏可见，主要操作固定在底部安全区。设备管理可保留为紧凑列表，但不应强行展示完整技术表格。

必须满足：

- 键盘可完成登录、切换 Tab、复制 ID、批准和撤销。
- 状态不可只依赖颜色，需带有文字和图标。
- 错误信息与倒计时使用 `aria-live` 提示。
- 短码、指纹和撤销确认支持屏幕阅读器完整朗读。
- Passkey 失败后提供安全的备用登录入口，不让用户丢失授权请求上下文。

## 10. 后端依赖

前端状态必须来自服务端 API，最小依赖如下：

```text
GET  /v1/me/agents?cursor=&limit=
POST /v1/agents
GET  /v1/agents/{agentId}
GET  /v1/agents/{agentId}/instances?cursor=&limit=
GET  /v1/approvals?status=pending&cursor=&limit=
GET  /v1/approvals/{requestId}
POST /v1/approvals/{requestId}/approve
POST /v1/approvals/{requestId}/deny
POST /v1/agents/{agentId}/instances/{instanceId}/revoke
GET  /v1/activity?cursor=&limit=

# Agent 公开属性
GET  /v1/public/agents?query=&tag=&verified=
GET  /v1/public/agents/{agentId}

# 登录用户管理 Agent 公开属性
GET  /v1/agents/{agentId}/public-profile
PATCH /v1/agents/{agentId}/public-profile
```

公开接口只返回经过公开策略过滤的 `PublicAgentProfile`，不能复用管理接口的完整 DTO。建议的数据结构如下：

```json
{
  "agentId": "did:agentid:agt_xxx",
  "name": "Research Assistant",
  "bio": "负责论文检索、摘要和资料整理",
  "attributes": [
    { "kind": "tag", "value": "academic", "label": "学术", "visibility": "public" },
    { "kind": "structured", "key": "skill", "value": "research", "label": "研究能力", "trust": "self_declared", "visibility": "public" },
    { "kind": "structured", "key": "project", "value": "open-research", "label": "项目", "visibility": "public" }
  ],
  "identityStatus": "verified",
  "presence": "unknown"
}
```

字段可见性规则：

| Agent 数据 | Agent 公开页 | 本人/Owner/Admin | Viewer | P2P 验证端 |
|---|---:|---:|---:|---:|
| AgentID、名称、公开简介 | 是 | 是 | 是 | 从 IBC 验证 |
| 公开标签和自声明能力 | 是 | 是 | 是 | 可读取 announce |
| 身份验证状态摘要 | 是 | 是 | 是 | 通过 IBC 验证 |
| 在线/最近活跃摘要 | 可选 | 是 | 是 | 否 |
| 未公开的 Agent 属性 | 否 | 是 | 按权限 | 否 |
| User ID、邮箱 | 否 | 本人可见 | 按策略 | 否 |
| InstanceID、PeerID、公钥指纹 | 否 | 是 | 按权限 | 消息中按需验证 |
| JTI、scope、到期时间 | 否 | Owner/Admin | 按权限 | 从 IBC 验证 |
| 完整 IBC | 否 | 不在网页直接展示 | 否 | 仅验证时使用 |
| 私钥、Cookie、设备码、验证码 | 否 | 否 | 否 | 否 |

批准、拒绝和撤销接口必须接受幂等键，并返回当前的权威状态。批准接口由服务端依据 `agent_members` 角色、请求有效期、实例公钥和 `agent_hint` 校验；前端不得根据本地缓存自行判断是否有授权或撤销权限。列表和详情接口返回显式的 `allowedActions`，仅用于呈现，不替代服务端授权判断。

## 11. 分期范围

### 一期

- Passkey 登录与恢复。
- 创建 Agent，单用户 Owner 模型。
- 二维码/短码设备授权。
- 设备列表、实例详情、续期提示和撤销。
- 审计记录。

### 二期

- 多成员角色与邀请。
- Agent 规则与高风险操作策略。
- 用户/组织归属声明的展示与验证。
- 跨组织 Agent 信任与共享。

### Agent 属性展示新增方案

建议拆成三个可独立验收的步骤：

1. **公开读取层**：增加 Agent 公开资料页和匿名只读 API；只返回公开 Agent 属性 DTO，不暴露管理接口字段。
2. **Owner 编辑层**：在 Agent 工作区编辑 Agent 公开简介、标签、能力和状态摘要开关，并提供访客预览。
3. **可信能力层**：后续引入独立的 Capability Credential，将“自声明能力”和“签发者验证能力”分开展示；第一版只实现自声明能力和 AgentID 身份验证。

Agent 公开资料页不应依赖 Demo Control Bridge；生产网页直接调用身份服务的公开只读 API。Demo 环境可以通过同一页面同时显示公开属性和本地 A/B 状态，但必须保持数据边界一致。

## 12. 验收标准

- 新用户无需理解技术 ID，即可创建 Agent 并完成一次设备授权。
- 已登录用户在网络正常且无需额外登录时，完成设备授权的中位时间不超过 60 秒。
- 用户可从任何主页面在两次点击内到达待授权请求或设备列表。
- 授权页面能在移动端单屏完成核对与批准。
- 撤销设备时，用户能明确知道被撤销的设备和后果。
- 同一授权请求被多个管理员同时处理时，最多签发一张 IBC，所有页面最终收敛到同一权威状态。
- 所有身份、授权、续期和撤销动作均有可筛选的审计记录。
- 任何页面都不展示实例私钥、完整设备码、长期网站令牌或网站内部 `userId`。
- 访客可以打开 Agent 公开资料页并看到该 Agent 主动公开的属性。
- 公开 API 无法通过参数或字段拼接读到 User ID、邮箱、JTI、完整 IBC 或实例私密信息。
- Owner/Admin 在登录后可以看到公开 Agent 属性，以及额外的实例、权限、JTI 和审计管理信息。
- 用户关闭某项 Agent 属性后，公开资料页和公开 API 在缓存失效后均不再返回该属性。
- 自声明能力和身份验证状态在界面上使用不同标签，不能把 AgentID 验证误显示为能力验证。
